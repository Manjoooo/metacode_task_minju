# 선생님께 검토받을 질문 정리

## 1. 먼저 설명할 구현 요약

선생님, 피드백 이후 현재 구현한 범위는 다음과 같습니다.

- Criteo 기반 광고 이벤트를 `impression`, `click`, `conversion`으로 변환
- Kafka Producer가 row 단위로 이벤트 발행
- Spark Structured Streaming으로 Bronze raw zone 적재
- Silver processed events, Gold campaign summary 구성
- Bronze/Silver/Gold를 Apache Iceberg table로 구성
- Iceberg `snapshots`, `files` metadata table 기반 health query 8개 생성
- Kafka, MinIO, Bronze/Silver/Gold 상태를 점검하는 health check script 구현
- Kafka down, MinIO down, Gold output missing 시나리오에서 alert/recovery 확인
- Iceberg `rewrite_data_files` compaction 수행
- 최종 compaction 자동화는 Windows Task Scheduler 기반 cron-style script로 구현
- MinIO/Spark SQL/HTML dashboard로 AWS S3/Glue/Athena/QuickSight 구조를 로컬에서 대체
- live replay dashboard로 near-real-time KPI 증가 화면 구성

현재 구현은 로컬 Docker 기반 PoC이며, K8s/EKS/Karpenter 기반 HA, Kafka multi-broker, Gold MERGE INTO, webhook 실전 연동은 향후 확장 방향으로 분리했습니다.

## 2. 선생님 피드백에 대해 반영한 부분

### 피드백 1. Airflow/K8s/HA 범위가 너무 넓고 모호함

반영 내용:

- Spark Structured Streaming job은 현재 Airflow가 직접 트리거하거나 재시작을 관리하는 구조가 아님을 README에 명시
- Airflow는 health check와 maintenance 자동화 실험에 사용했다고 구분
- compaction은 Airflow 권한 문제로 인해 최종적으로 Task Scheduler 기반 cron-style script로 구현했다고 명시
- K8s/EKS/Karpenter는 실제 구현 완료가 아니라 향후 확장 설계로 분리
- K8s 도입만으로 HA가 자동 구성되는 것이 아니라, driver restart policy, checkpoint, node pool, resource request/limit, Kafka replication까지 함께 설계해야 한다고 추가

선생님께 질문:

> 현재처럼 로컬 Docker 기반 PoC에서는 Spark Streaming을 직접 실행하고, Airflow/K8s 기반 자동 재시작은 향후 확장 설계로 분리해도 괜찮을까요?

---

### 피드백 2. 장애 시나리오와 요구사항 구현이 섞여 있음

반영 내용:

- README에서 실제 장애성 시나리오와 단순 검증 항목을 구분
- 실제 장애로 검증한 항목: Kafka broker down, MinIO storage down, Gold output missing, compaction failure
- 요구사항 검증 항목: row count, Iceberg table list, snapshot/files metadata query, live dashboard

선생님께 질문:

> 현재 Kafka/MinIO/Gold missing 중심의 장애 시나리오 외에, 제출 전에 추가하면 좋은 장애 시나리오가 있을까요?

---

### 피드백 3. 로그 파일을 얼마나 유지할지 고민 필요

반영 내용:

- README에 log retention 정책 추가
- active alert log는 최근 7~30일 유지
- maintenance report는 최근 30일 유지
- 오래된 로그는 object storage archive 또는 삭제
- 중요한 장애 로그는 별도 incident report로 보존하는 방향 추가

선생님께 질문:

> README에 log retention 정책을 설계로 추가했는데, 실제 rotation script까지 구현하는 것이 필요할까요?

---

### 피드백 4. 파일 로그보다 webhook alert interface가 좋음

반영 내용:

- `AlertSender` 구조를 README에 설명
- 현재 기본 구현은 FileAlertSender 기반 `alerts.log`
- 향후 Slack/Discord/Teams webhook을 붙일 수 있는 WebhookAlertSender, MultiAlertSender 구조로 확장 가능하다고 정리

선생님께 질문:

> 현재는 파일 로그 기반 PoC이고 AlertSender 인터페이스만 구성했습니다. 실제 Discord/Slack webhook까지 붙여보는 것이 좋을까요, 아니면 현재 구현에서는 인터페이스 설계와 파일 로그 증거로 충분할까요?

## 3. 최종적으로 물어볼 핵심 질문

1. 현재 로컬 Docker 기반 구현 범위가 필수 과제 조건을 충족하는지 확인받고 싶습니다.
2. AWS 직접 사용 대신 MinIO/Spark SQL/HTML dashboard로 대체한 표현이 적절한지 확인받고 싶습니다.
3. Spark Streaming을 Airflow가 직접 트리거하지 않고, 로컬 streaming job으로 실행한 현재 구조가 제출 범위에서 충분한지 확인받고 싶습니다.
4. Airflow compaction 실패 후 Task Scheduler로 전환한 방식이 cron-style maintenance 자동화로 인정 가능한지 확인받고 싶습니다.
5. Kafka/MinIO/Gold missing 이외에 추가하면 좋은 장애 시나리오가 있을지 조언받고 싶습니다.
6. 파일 기반 alert log와 AlertSender 구조를 제시했는데, 실제 webhook까지 구현하는 것이 필요한지 확인받고 싶습니다.
7. log retention 정책은 README 설계로 충분한지, 실제 rotation script를 추가하는 것이 좋은지 확인받고 싶습니다.
8. 남은 시간에 OpenRTB/request topic을 추가하는 것보다 현재 구현의 안정성과 설명을 강화하는 것이 나은지 확인받고 싶습니다.
9. Gold summary를 현재 batch summary로 두고, late conversion MERGE INTO는 향후 확장으로 제시해도 되는지 확인받고 싶습니다.
10. 발표에서는 end-to-end 실행 결과, Iceberg metadata health query, compaction 자동화, 장애 감지/복구를 중심으로 강조해도 되는지 조언받고 싶습니다.
